# 赛题规则与 Agent 执行约束

> 适用项目：面向噪声标签数据的细粒度图像识别鲁棒微调  
> 建议文件位置：仓库根目录 `COMPETITION_RULES.md`  
> 用途：供代码 Agent、实验 Agent、评审 Agent 在设计方案、修改代码、运行实验和生成提交文件前读取。  
> 规则来源：赛事官方赛题说明 PDF。若本文件与组委会后续正式通知冲突，以最新正式通知为准。

---

## 1. 任务定义

本赛题是**含噪标签条件下的单标签细粒度图像分类任务**。

核心目标：

- 使用赛事官方提供的含噪训练集训练或微调模型；
- 识别自然动植物细粒度类别；
- 在人工精确标注、类别分布均衡的测试集上取得较高的 **Top-1 Accuracy**；
- 在噪声标签、细粒度类别差异和后续阶段长尾分布下保持泛化性与稳定性。

线上主要指标：

```text
Accuracy = 预测正确的测试样本数 / 测试样本总数
```

每张测试图像只有一个真值标签。

## 2. Agent 必须遵守的最高优先级硬约束

以下规则属于不可突破的竞赛边界。任何实验方案、代码修改或提交方案只要违反其中一项，就不能作为正式参赛方案。

### RULE-01：骨干网络固定

必须使用：

```text
CLIP ViT-B/32
```

禁止：

- 替换为其他 CLIP 规模；
- 使用 ViT-B/16、ViT-L、ViT-H 等更大或不同骨干；
- 使用 DINO、DINOv2、MAE、Swin、ConvNeXt、SigLIP、EVA、BLIP 等其他视觉基础模型替代核心骨干；
- 使用商业闭源视觉模型 API 或在线大模型推理接口替代核心识别流程。

Agent 在新增模型配置、下载权重或修改 backbone 时，必须先验证最终骨干仍然是 CLIP ViT-B/32。

### RULE-02：预训练权重来源受限

允许使用 CLIP ViT-B/32 的公开预训练权重，但仅限 OpenAI 官方公开的对应权重，例如：

- `openai/clip` 中的 ViT-B/32；
- Hugging Face Transformers 中与 OpenAI 官方 CLIP ViT-B/32 对应的版本。

禁止使用：

- 在额外数据集上自行训练或继续预训练的 ViT-B/32 权重；
- 第三方使用私有数据或额外数据微调后的权重；
- 其他比赛、其他阶段或外部任务训练出的权重；
- 来源无法确认的 checkpoint。

Agent 引入 checkpoint 时必须记录：

```yaml
backbone: CLIP ViT-B/32
weight_source: OpenAI official pretrained weights
library:
revision_or_commit:
download_identifier:
```

### RULE-03：禁止外部数据

每个比赛阶段只能使用**当前阶段官方提供的数据**。

禁止在训练过程中引入任何形式的额外数据，包括：

- 公开数据集；
- 私有数据集；
- 网络爬取图像；
- 人工补充标注数据；
- 类别百科图片；
- 其他比赛数据；
- 其他阶段的赛事数据；
- 由外部数据训练得到的教师模型、分类器、原型或缓存特征。

跨阶段限制：

- 复赛不得使用初赛数据；
- 半决赛不得使用初赛或复赛数据；
- 当前阶段的数据和产物不得默认迁移到下一阶段继续训练。

允许迁移的是**算法、代码和超参数经验**，不是上一阶段的数据、样本特征、伪标签、类别原型或基于上一阶段数据训练的参数。

### RULE-04：测试集禁止参与训练

测试数据仅可用于最终预测和提交，不得以任何方式参与训练。

禁止行为包括但不限于：

- 将测试图像加入训练 DataLoader；
- 对测试集执行自监督训练；
- 使用测试图像进行无监督域适配；
- 根据测试图像聚类结果调整模型；
- 使用测试分布生成类别原型；
- 使用测试时训练、测试时适配或梯度更新；
- 根据测试图像的预测分布反复调整训练超参数；
- 人工浏览测试图像后针对性修改类别映射或数据策略。

允许的测试处理仅限于**无参数学习的确定性推理流程**，例如 resize、crop、normalize、单模型前向推理及合规的结果格式化。

> 注意：是否允许测试时增强（TTA）在官方说明中未被明确禁止，但必须满足“单一模型或单一推理流程”及“测试数据不参与训练”。在未获得组委会明确答复前，正式方案应谨慎使用，并记录所有增强与聚合方式。

### RULE-05：禁止多模型集成

最终提交不得采用：

- 多模型 ensemble；
- 多个独立 checkpoint 投票；
- 不同随机种子模型平均；
- 不同 backbone 融合；
- 多个分类头独立训练后融合；
- 多模型 logits/概率加权；
- 多个独立训练流程的预测结果后处理融合。

最终评测结果必须来自：

```text
单一模型
或
单一推理流程
```

为降低规则风险，本项目默认采用：

```text
一个训练完成的 checkpoint
+ 一个确定的推理脚本
+ 一份 pred_results.csv
```

以下情况必须标记为规则风险并在正式提交前确认：

- EMA 权重是否视为同一模型；
- SWA/权重平均是否属于模型融合；
- checkpoint soup；
- 多折交叉验证模型融合；
- 多 seed 平均；
- 多 prompt 模板集成；
- 多 crop、多尺度预测聚合。

Agent 不得擅自把上述方法加入正式提交路径。

### RULE-06：训练流程必须自动、完整、可复现

最终训练过程必须能通过提交代码直接复现，包括：

- 数据索引构建；
- 训练集/验证集划分；
- 噪声筛选；
- 样本打分；
- 样本重加权；
- 伪标签生成；
- 课程学习；
- checkpoint 选择；
- 推理；
- CSV 生成。

人工数据清洗只能用于开发分析，不能成为正式训练的必要前置步骤。

禁止：

- 手工删除某些图片后再训练；
- 人工查看样本并维护黑名单；
- 手工修改错误标签作为正式训练输入；
- 使用未纳入脚本的本地中间文件；
- 依赖无法重建的缓存；
- 只提交最终 checkpoint 而无法复现训练过程。

## 3. 允许研究和实现的方向

在遵守硬约束的前提下，可在 CLIP ViT-B/32 上探索：

### 3.1 参数高效微调

- Linear probing；
- Prompt Tuning；
- Adapter；
- LoRA；
- 部分层解冻；
- BitFit；
- 可学习分类头；
- Cosine classifier；
- 其他不改变固定骨干身份的参数高效方法。

### 3.2 噪声标签鲁棒学习

- Label Smoothing；
- Generalized Cross Entropy；
- Symmetric Cross Entropy；
- Bootstrap loss；
- Focal 类损失；
- 样本置信度加权；
- small-loss selection；
- 动态样本过滤；
- curriculum learning；
- co-teaching 思想的单模型化实现；
- 自动伪标签修正；
- 软标签训练；
- 类别原型一致性约束。

### 3.3 预训练知识保护

- 冻结大部分 backbone；
- 小学习率解冻；
- L2-SP；
- feature distillation；
- logit distillation；
- 表征漂移约束；
- 与初始 CLIP 特征保持一致；
- 教师—学生机制，但教师不得依赖外部数据或违规模型。

### 3.4 长尾处理

复赛和半决赛可考虑：

- class-balanced sampling；
- reweighting；
- logit adjustment；
- balanced softmax；
- decoupled classifier training；
- tail-aware margin；
- 分阶段训练。

任何长尾方法仍需遵守“仅使用当前阶段官方数据”。

## 4. 数据集信息

| 阶段 | 类别数 | 训练样本数 | 测试样本数 | 标签噪声 | 长尾分布 |
|---|---:|---:|---:|---|---|
| 初赛 | 500 | 103,218 | 24,967 | 有 | 官方未强调 |
| 复赛 | 1,500 | 297,282 | 74,896 | 有 | 有 |
| 半决赛 | 1,000 | 180,274 | 49,857 | 有 | 有 |

共同特征：

- 类别以自然动植物为主；
- 单标签分类；
- 训练数据按类别文件夹组织；
- 测试标签不公开；
- 部分类别视觉差异很小；
- 训练标签可能包含错误标注、弱相关标注和难样本干扰；
- 部分图片在文件系统中可能显示异常或存在截断，但官方说明其可被 Pillow 正常读取。

Agent 实现数据加载时应：

1. 默认使用 Pillow 读取；
2. 对 `OSError`、截断图像、损坏文件增加显式日志；
3. 不得仅因系统预览失败就删除图片；
4. 不得静默跳过大量样本；
5. 输出读取成功率与失败样本清单；
6. 若启用 `ImageFile.LOAD_TRUNCATED_IMAGES = True`，需在运行日志和配置中记录；
7. 验证训练、验证和测试的样本数量与官方文件清单一致。

## 5. 阶段规则与成绩构成

### 5.1 初赛

作用：

- 熟悉任务；
- 验证训练和提交流程；
- 完成晋级。

初赛成绩不计入最终线上综合成绩。

### 5.2 复赛与半决赛

线上综合成绩：

```text
线上综合成绩 = 复赛成绩 × 40% + 半决赛成绩 × 60%
```

因此，Agent 不应只针对初赛数据特征过拟合算法结构，应优先保留：

- 跨类别规模扩展能力；
- 对不同噪声水平的稳定性；
- 长尾兼容性；
- 可配置的数据路径和类别数量；
- 可复用但不携带上一阶段数据的训练流程。

### 5.3 总决赛

总决赛采用：

- 客观评测；
- 线下答辩；
- 复现与代码复核。

重点考察：

- 方法创新性；
- 技术完整性；
- 复现质量；
- 实验分析；
- 表达展示。

若提交结果低于官方基线，赛事方有权认定为无效成绩。

## 6. 正式提交文件规则

### 6.1 预测文件

文件名必须为：

```text
pred_results.csv
```

随后压缩为 ZIP 文件提交。

每行包含两个字段：

```text
图片文件名, 四位类别编号
```

示例：

```csv
xxxxxxxxxxxx.jpg,0001
xxxxxxxxxxxy.jpg,0123
xxxxxxxxxxxz.jpg,0456
```

### 6.2 类别编号

类别编号必须：

- 使用四位数字；
- 不足四位时在左侧补 `0`；
- 不得输出整数形式导致前导零丢失；
- 不得输出类别名称；
- 不得输出概率；
- 不得额外输出表头，除非官方平台模板明确要求表头。

建议在代码中使用：

```python
label_str = f"{class_id:04d}"
```

### 6.3 文件名一致性

测试图片文件名必须与官方测试集完全一致，包括：

- 大小写；
- 扩展名；
- 字符；
- 顺序对应关系。

禁止：

- 使用完整绝对路径；
- 修改扩展名；
- 重命名图片；
- 重复图片；
- 遗漏图片；
- 添加不在测试集中的图片。

### 6.4 CSV 生成后的强制校验

Agent 生成提交文件后必须执行：

```text
[ ] 文件名为 pred_results.csv
[ ] 行数等于官方测试图片数
[ ] 每个测试文件恰好出现一次
[ ] 不存在额外文件名
[ ] 不存在重复文件名
[ ] 标签满足 ^\d{4}$
[ ] 标签范围合法
[ ] 无 NaN、空行和空标签
[ ] 无意外索引列
[ ] 编码和分隔符可被标准 CSV 工具读取
[ ] ZIP 中包含正确的 pred_results.csv
```

## 8. Agent 的代码修改原则

### 8.1 修改前检查

Agent 在实现新方法前必须回答：

1. 是否仍使用 CLIP ViT-B/32？
2. 权重是否来自 OpenAI 官方公开权重？
3. 是否引入了外部数据或外部数据训练产物？
4. 是否让测试集参与了任何参数更新、统计拟合或策略选择？
5. 是否形成多模型集成？
6. 是否能从原始当前阶段数据自动复现？
7. 是否能通过单一推理流程生成合规 CSV？
8. 是否会把上一阶段数据或产物带入当前阶段？

只要任一项不确定，必须先标记：

```text
RULE_RISK
```

不得直接并入正式提交分支。

### 8.2 配置优先

所有实验变量应进入配置文件或命令行参数，不得散落硬编码：

- backbone 名称；
- pretrained 权重来源；
- 解冻层；
- loss 类型；
- loss 超参数；
- 数据划分随机种子；
- batch size；
- epoch；
- learning rate；
- weight decay；
- augmentation；
- 样本筛选阈值；
- 类别数量；
- 数据路径；
- 输出路径。

### 8.3 实验可追踪性

每个实验至少记录：

```yaml
experiment_id:
stage:
git_commit:
backbone:
pretrained_source:
dataset_root:
split_file:
seed:
train_samples:
val_samples:
num_classes:
method:
loss:
optimizer:
scheduler:
epochs:
best_metric:
checkpoint:
submission_file:
rule_compliance:
```

### 8.4 不得自动删除或覆盖关键产物

Agent 不得未经确认：

- 删除训练日志；
- 覆盖最佳 checkpoint；
- 修改已有实验结果；
- 重新生成 split 并覆盖旧 split；
- 把验证集样本重新并入训练集；
- 清理失败实验的证据；
- 使用同名实验目录覆盖已有结果。

## 9. 数据划分与验证集规则

官方未提供公开测试标签，因此项目需要从当前阶段训练集内构造验证集。

Agent 应遵守：

- 验证集仅从当前阶段官方训练数据划分；
- 划分脚本必须可复现；
- 固定随机种子；
- 保存明确的 train/val manifest；
- 不根据测试集表现修改划分；
- 不将同一文件同时放入训练集和验证集；
- 若进行去重，应只在当前阶段官方训练数据内部自动执行；
- 去重规则、被移除样本和哈希方法必须可复现；
- 不能人工查看验证错误后手工修改样本标签作为正式方案。

推荐至少保留：

```text
splits/
  train.csv
  val.csv
  split_metadata.json
```

`split_metadata.json` 应记录：

```json
{
  "seed": 42,
  "strategy": "stratified",
  "train_count": 0,
  "val_count": 0,
  "num_classes": 0,
  "source": "current-stage official training set only"
}
```

## 10. 单模型与单一推理流程的项目解释

为避免 Agent 对规则做宽松解释，本项目默认采用以下保守标准：

### 合规默认

- 一次训练流程；
- 一个最终 checkpoint；
- 一个分类模型；
- 一个推理入口；
- 一套确定的预处理；
- 一次输出类别；
- 不融合其他独立模型结果。

### 需要规则确认

以下方法可能处于灰区，不得默认用于正式提交：

- EMA 与原模型预测融合；
- SWA；
- 权重平均；
- model soup；
- 多 prompt 模板平均；
- 多 crop TTA；
- 多尺度预测；
- checkpoint averaging；
- k-fold 模型集成；
- 不同 seed 模型融合；
- teacher 与 student 同时参与最终预测；
- 将多个分类头预测求平均。

Agent 可以将灰区方法实现为研究分支，但必须：

1. 默认关闭；
2. 配置名包含 `experimental` 或 `rule_risk`；
3. 不进入正式 submission pipeline；
4. 在得到组委会明确答复后再启用。

## 11. 明确禁止的高风险做法

Agent 不得提出或执行：

- 使用测试集做自监督预训练；
- 使用测试集做 BatchNorm/统计量拟合；
- 使用测试预测分布优化类别先验；
- 下载类别对应网络图片；
- 使用 iNaturalist、ImageNet 或其他外部数据；
- 使用外部模型给训练集重新标注；
- 用 GPT、闭源视觉 API 或在线识图接口生成标签；
- 人工查看并修改训练标签后作为最终数据；
- 融合多个 checkpoint；
- 使用非 OpenAI 官方 CLIP ViT-B/32 预训练权重；
- 将初赛训练结果继续用于复赛；
- 将复赛 checkpoint 用于半决赛；
- 把赛事数据上传到公开仓库、网盘或无权限控制的平台；
- 将赛事数据用于与本比赛无关的实验或发布。

## 12. 数据保密与知识产权

赛事数据：

- 仅限本次比赛使用；
- 严禁泄露；
- 严禁传播；
- 严禁用于与赛事无关的用途；
- 不应提交到公开 Git 仓库；
- 不应写入 Docker 镜像；
- 不应出现在日志样例、Issue 附件或公开实验平台。

项目代码和文档必须避免：

- 提交真实数据文件；
- 提交测试图片；
- 提交包含赛事图片的可视化；

## 17. 当前项目的保守执行口径

在组委会没有进一步解释前，本项目采用以下保守策略：

1. 不使用测试时训练或测试时适配；
2. 不使用多模型、多 seed、多折集成；
3. 不融合多个 checkpoint；
4. 不使用外部类别文本描述生成额外监督，除非文本仅来自当前阶段官方提供的类别信息且确认合规；
5. 不使用外部视觉模型进行样本清洗或伪标签；
6. 不跨阶段复用数据、特征、原型、伪标签或模型参数；
7. 只保留一个最终 checkpoint；
8. 所有清洗、筛选、重加权必须由代码自动完成；
9. 所有规则灰区实验必须与正式提交路径隔离。

## 18. 尚需向组委会确认的规则灰区

官方文件没有完全明确以下细节。若项目计划使用，应先咨询组委会并保存答复：

1. 多 crop 或多尺度 TTA 是否属于“单一推理流程”；
2. 多 prompt 模板平均是否构成集成；
3. EMA、SWA、checkpoint averaging 是否构成模型融合；
4. 是否允许只使用官方类别名称构造文本 prompt；
5. 是否允许使用公开词典或知识库扩充类别描述；
6. 是否允许用 OpenAI CLIP 文本编码器生成类别文本原型；
7. 是否允许同一模型的多个视角预测平均；
8. 是否允许当前阶段训练集内部的自动去重；
9. 是否允许当前阶段训练集内部进行自监督预训练；
10. 是否允许使用当前阶段训练数据训练 teacher，再训练最终单一 student；
11. 是否允许伪标签替换原始标签；
12. 是否允许将初赛确定的超参数和代码直接用于复赛、半决赛；
13. 最终提交 CSV 是否带表头；
14. ZIP 内是否只能有 `pred_results.csv`；
15. 类别编号是从 0 还是从 1 开始，以官方类别映射文件为准。

在未确认前，Agent 应采用最保守解释。
